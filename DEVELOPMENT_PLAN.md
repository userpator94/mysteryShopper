# План разработки: роли и кабинет заказчика

Цель: дать заказчикам возможность регистрироваться и создавать задачи (предложения). Исполнители (тайные покупатели) продолжают пользоваться текущим флоу.

---

## Общая архитектура

- **Роли:** `user` (исполнитель), `employer` (заказчик).
- **Единые** `signup` / `login` с полем `role` и опциональными полями заказчика (компания и т.д.).
- **Backend** проверяет роль и права на все операции с офферами и заявками.

---

## Требования к API

Связь с БД: данные берутся из таблиц **users**, **employers**, **offers**. Поля таблиц — ниже; в API используются те же имена или указанные алиасы.

### Справочник таблиц (кратко)

| Таблица    | Ключевые поля |
|-----------|----------------|
| **users** | id (uuid), name, surname, email, phone, role, password_hash, is_active, created_at, updated_at |
| **employers** | id (uuid), user_id (FK → users.id), company, description, website, is_active, created_at, updated_at |
| **offers** | id (uuid), image_id, title, numeric_info, tags, is_promo, employer_id (FK → employers.id), start_date, end_date, is_active, price, max_participants, current_participants, description, requirements, location, created_at, updated_at |

Роль в **users**: `user` (исполнитель) или `employer` (заказчик). У заказчика должна быть запись в **employers** с `user_id = users.id`.

---

### Аутентификация

| Метод | Эндпоинт | Кто | Тело запроса | Ответ |
|-------|----------|-----|--------------|--------|
| POST | `/api/login` | Все | `{ "email", "password" }` | 200: `{ success, data: { token, user: { id, name, surname, email, phone, role }, expiresIn } }`. 401 при неверных данных. |
| POST | `/api/signup` | Все | `{ name, surname, email, phone, password, role?, company?, description?, website? }` | 200: тот же формат, что и login. Для `role: "employer"` обязательны или ожидаемы: company; опционально description, website. При успехе создаётся запись в **users** и при `role === "employer"` — в **employers** (user_id, company, description, website). 4xx при валидации (дубликат email и т.д.). |
| GET  | `/api/me`    | Авторизованный | — | 200: `{ success, data: { id, name, surname, email, phone, role, company?, description?, website? } }`. Для employer поля company/description/website брать из **employers** по user_id. 401 без токена. |

Заголовок авторизации: `Authorization: Bearer <token>`.

---

### Офферы (задачи)

| Метод | Эндпоинт | Кто | Тело / параметры | Ответ |
|-------|----------|-----|-------------------|--------|
| GET | `/api/offers` | Публично или с токеном | Query: query, category, minPrice, maxPrice, minRating, sortBy, sortOrder, page, limit | 200: `{ success, data: Offer[] }`. Поля оффера — см. ниже. |
| GET | `/api/offers/:id` | Авторизованный | — | 200: `{ success, data: Offer }`. 404 если нет. |
| POST | `/api/offers` | Только employer | Body: title, description, price, location, requirements, tags, start_date, end_date, max_participants, is_promo?, image_id? | 201: `{ success, data: Offer }`. **employer_id** не принимается от клиента: определить из JWT (user_id → employers.id). 403 если роль не employer. 422 при ошибках валидации. |
| GET | `/api/my/offers` или `/api/employer/offers` | Только employer | Query: page, limit (опц.) | 200: `{ success, data: Offer[] }` — только офферы текущего заказчика (employer_id = employers.id по user_id из JWT). 403 если не employer. |
| PATCH | `/api/offers/:id` | Владелец оффера (employer) | Body: часть полей (title, description, price, location, requirements, tags, start_date, end_date, max_participants, is_promo, is_active) | 200: `{ success, data: Offer }`. 403 если не владелец. 404 если оффер не найден. 422 при валидации. |
| DELETE | `/api/offers/:id` | Владелец оффера | — | 200/204 без тела или 200 с `{ success, message }`. Либо мягкое удаление: PATCH с `is_active: false`. 403 если не владелец. 404 если не найден. |

Формат объекта **Offer** в ответах (поля соответствуют **offers** + джойн с employers/users для имён):

- id, image_id, title, numeric_info, tags, is_promo, employer_id, start_date, end_date, is_active, price, max_participants, current_participants, description, requirements, location, created_at, updated_at  
- employer_name, employer_surname, employer_company (из **users** + **employers**: u.name, u.surname, e.company)  
- image_url, image_alt_text (из **images** по image_id, если есть)  
- available_slots: max_participants - current_participants (опционально, можно считать на клиенте)

Типы: id/uuid — строки; даты — ISO 8601; price — число или строка с числом; max_participants, current_participants — число.

---

### Заявки (applications)

| Метод | Эндпоинт | Кто | Тело / параметры | Ответ |
|-------|----------|-----|-------------------|--------|
| GET | `/api/offers/:id/applications` | Владелец оффера (employer) | — | 200: `{ success, data: Application[] }`. 403 если не владелец. 404 если оффер не найден. |
| PATCH | `/api/applications/:id` | Владелец оффера | Body: `{ status: "approved" \| "rejected" }` | 200: `{ success, data: Application }`. 403 если не владелец. 404 если заявка не найдена. 422 при неверном status. |

(Структура таблицы заявок и поля Application оставить по текущей реализации БД.)

---

### Общие правила

- **401 Unauthorized** — нет или невалидный/истёкший токен. Тело: `{ success: false, error: { message, code? } }`.
- **403 Forbidden** — недостаточно прав (роль или не владелец ресурса). Тело: `{ success: false, error: { message, code? } }`.
- **404 Not Found** — ресурс не найден. Тело: `{ success: false, error: { message } }`.
- **422 Unprocessable Entity** — ошибка валидации. Тело: `{ success: false, error: { message, code?, field?, errors?: [{ field, message }] } }`.
- Успешные ответы: `{ success: true, data: ... }`. Пагинация (если есть): `{ success: true, data: [], meta: { page, limit, total } }`.

## Порядок внедрения и зависимости

```
Фаза 0 (контракт) 
    ↓
BE Фаза 1 (роли, auth)  →  FE Фаза 1 (роль, навигация)
    ↓                           ↓
BE Фаза 2 (CRUD офферов) →  FE Фаза 2 (регистрация заказчика)
    ↓                       FE Фаза 3 (кабинет, создание/редактирование)
BE Фаза 3 (заявки)       →  FE Фаза 4 (защита маршрутов, профиль)
    ↓                       FE Фаза 5 (заявки в кабинете) — опционально
```

- **Сначала BE:** роли и эндпоинты создания/списка/редактирования офферов.
- **Параллельно после BE Фазы 1:** FE Фаза 1 (роль, разная навигация и редирект после логина).
- **После BE Фазы 2:** FE Фазы 2–3 (регистрация заказчика + кабинет и форма создания/редактирования).
- **Далее:** FE Фаза 4, затем при необходимости BE и FE Фаза 5.

## Критерии готовности (чек-лист)

- [ ] Заказчик может зарегистрироваться (выбор роли + компания) и войти.
- [ ] После входа заказчик видит навигацию «Мои задачи» / «Создать» и не видит «Избранное» (или по продукту).
- [ ] Заказчик создаёт задачу (оффер) через форму и видит её в списке «Мои задачи».
- [ ] Заказчик может отредактировать и удалить/деактивировать свою задачу.
- [ ] Исполнитель по-прежнему видит список офферов, детали, заявки, отчёты; создание офферов ему недоступно.
- [ ] В ответах auth и в `GET /me` возвращается `role`; фронт сохраняет и использует роль для UI и редиректов.
